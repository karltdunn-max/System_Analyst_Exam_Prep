REST API Concepts

Think of a REST API as a very disciplined waiter in a restaurant: you give a clear order (HTTP method + URL), they bring back exactly what you asked for (status code + JSON/XML), and they never “remember” anything between visits (stateless). If you can explain it that simply, you can teach it to anyone.

Core Ideas

HTTP Methods Status Codes Idempotency JSON vs XML Swagger / OpenAPI
  • REST is a style of designing APIs around resources (nouns) and standard HTTP verbs.
  • Resources are things: /users, /accounts/123, /orders/987/items.
  • Clients talk to servers using HTTP methods: GET, POST, PUT, PATCH, DELETE.
  • Every response carries a status code that tells you what happened.
  • REST is stateless: the server doesn’t remember previous calls — each request is complete on its own.
Example – A simple REST call
GET /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "name": "Karl Dunn",
  "role": "Systems Analyst"
}

You asked for user 42 with a GET. The server replied with 200 OK and a JSON body. No session, no magic — just a clean request/response contract.

Methods, Status Codes & Idempotency

  • GET – Read data. Idempotent.
  • POST – Create something new. Not idempotent.
  • PUT – Replace a resource. Idempotent.
  • PATCH – Partially update. Usually idempotent if designed well.
  • DELETE – Remove a resource. Idempotent.
  • 200 OK – Success.
  • 201 Created – New resource created (POST).
  • 204 No Content – Success, nothing to return.
  • 400 Bad Request – Your input is wrong.
  • 401 Unauthorized – No valid credentials.
  • 403 Forbidden – You’re authenticated but not allowed.
  • 404 Not Found – Resource doesn’t exist.
  • 500 Internal Server Error – The server broke.
Example – Idempotency in practice
PUT /users/42/email
{
  "email": "karl@example.com"
}

Call this once or ten times — the user’s email ends up the same. That’s idempotency: repeated calls don’t change the final state.

Swagger / OpenAPI & Troubleshooting

Swagger (OpenAPI) is like the blueprint and menu for your API: it tells you what endpoints exist, what they expect, and what they return. A good Systems Analyst can read Swagger like a map and spot when reality doesn’t match the contract.

Example – Reading Swagger
POST /orders
Request body:
{
  "userId": "string",
  "items": [
    { "sku": "string", "quantity": 1 }
  ]
}

Responses:
201 Created - Order created
400 Bad Request - Validation error

From this alone, you know how to test the API, what fields are required, and what success vs failure looks like. That’s enough to design test cases and debug issues.

Questions & Answers – REST APIs
Q: Why is POST not idempotent?
A: Because calling POST multiple times usually creates multiple resources (e.g., multiple orders), changing the state each time.
Q: What’s the difference between 401 and 403?
A: 401 means “you’re not properly authenticated”; 403 means “you are authenticated, but you’re not allowed to do this.”
Q: An API returns 200 but the JSON is missing fields. What do you do?
A: Reproduce in Swagger, compare the response to the contract, check recent deployments and logs, and work with engineering to fix either the implementation or the documentation.

2. Which status code is most appropriate for a successful resource creation via POST?

3. A client calls an API without any authentication token. Which status code fits best?

4. Which format is typically lighter and easier to parse in modern REST APIs?

5. What is the main purpose of Swagger/OpenAPI?

SQL – Queries, Joins, Aggregations & Performance

SQL is how you ask questions of your data. A good Systems Analyst doesn’t just write queries — they can explain what a query is doing, why it’s slow, and how to fix it without breaking the business logic.

Joins & Aggregations

INNER / LEFT JOIN GROUP BY HAVING Subqueries

Joins answer: “How do these tables relate?” Aggregations answer: “What’s the summary?”

Example – INNER vs LEFT JOIN
-- All users who have at least one order
SELECT u.id, u.name, COUNT(o.id) AS order_count
FROM users u
INNER JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;

-- All users, even if they have no orders
SELECT u.id, u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;

The first query only shows users with orders. The second shows everyone, with 0 for users who haven’t ordered yet. That’s the practical difference between INNER and LEFT JOIN.

Example – GROUP BY & HAVING
-- Departments with more than 10 employees
SELECT dept, COUNT(*) AS employee_count
FROM employees
GROUP BY dept
HAVING COUNT(*) > 10;

WHERE filters rows before grouping; HAVING filters groups after aggregation. That’s a classic exam trick.

Subqueries & Window Functions

Subqueries let you ask “meta‑questions” (e.g., “give me users whose total spend is above average”). Window functions let you rank, compare, and analyze rows without collapsing them into a single group.

Example – Correlated Subquery
-- Users whose total order amount exceeds $1,000
SELECT u.id, u.name
FROM users u
WHERE (
  SELECT SUM(o.amount)
  FROM orders o
  WHERE o.user_id = u.id
) > 1000;

The subquery depends on each row of users. That’s a correlated subquery — powerful but can be slower on large datasets.

Example – Window Function
-- Rank employees by salary within each department
SELECT
  dept,
  name,
  salary,
  RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS dept_salary_rank
FROM employees;

PARTITION BY splits the data into groups (departments), and RANK() orders people inside each group. You keep all rows, but now each row knows where it stands.

Performance, NULLs & Troubleshooting

  • SELECT * is convenient but often wasteful — it pulls more data than needed.
  • Indexes speed up reads but slow down writes; use them where lookups matter.
  • Functions on indexed columns (e.g., LOWER(email)) can prevent index usage.
  • NULL is “unknown”, not zero and not an empty string.
  • WHERE column = NULL never matches; use IS NULL instead.
Example – Slow Query Fix
-- Slow: function on indexed column
SELECT * FROM users
WHERE LOWER(email) = 'karl@example.com';

-- Faster: normalize input, use index
SELECT * FROM users
WHERE email = 'karl@example.com';

The first query forces a function on every row, often bypassing the index. The second lets the database use the index directly.

Questions & Answers – SQL
Q: Why did my report suddenly show more rows after adding a JOIN?
A: The JOIN created duplicates (e.g., one user with many orders). You may need DISTINCT, aggregation, or a different join strategy.
Q: What’s the difference between WHERE and HAVING?
A: WHERE filters individual rows before grouping; HAVING filters aggregated groups after GROUP BY.
Q: Why does WHERE column = NULL return no rows?
A: Because NULL means “unknown”; comparisons with NULL are neither true nor false. Use IS NULL instead.
SQL Quiz (5 Questions)

1. Which JOIN returns all rows from the left table and matching rows from the right?

2. Which clause filters aggregated results after GROUP BY?

3. How do you correctly check for NULL values in a column?

4. In a correlated subquery, how often is the subquery executed?

5. Which of the following can prevent an index from being used efficiently?

AWS – EC2, S3, IAM, VPC & CloudWatch

AWS is just rented infrastructure with really good knobs. As a Systems Analyst, you don’t need to build the engine — but you do need to know what each knob does and which one to turn when something breaks.

Core Services

EC2 S3 IAM VPC CloudWatch
  • EC2 – Virtual machines. Where your Java/Spring apps and APIs often run.
  • S3 – Object storage. Great for logs, backups, and large files.
  • IAM – Identity & Access Management. Who can do what.
  • VPC – Virtual network. Subnets, routing, isolation.
  • CloudWatch – Metrics and logs. Your first stop when something misbehaves.
Example – EC2‑hosted API
Client --(HTTPS)--> Load Balancer --> EC2 instance (Spring Boot API)
                         |
                         +-- CloudWatch logs & metrics
                         |
                         +-- IAM role for S3 / other services

When the API fails, you think in layers: network (VPC, security groups), instance health (EC2), app logs (CloudWatch), and permissions (IAM).

Security Groups, NACLs & Storage Types

  • Security Group – Instance‑level firewall, stateful.
  • NACL – Subnet‑level firewall, stateless.
  • S3 – Object storage (buckets, objects).
  • EBS – Block storage for EC2 root and data volumes.
  • EFS – Shared file system for multiple EC2 instances.
Example – S3 Access Denied
Symptom:
  Application on EC2 fails to read from S3 with "AccessDenied".

Checklist:
  1. Does the EC2 instance have an IAM role?
  2. Does that role's policy allow s3:GetObject on the bucket?
  3. Does the bucket policy allow access from that role/account?
  4. Is there any explicit deny in the policy?

Most S3 issues are IAM or bucket policy problems, not “S3 is down.” Knowing that saves hours.

CloudWatch & Troubleshooting

  • CloudWatch collects metrics (CPU, memory, latency) and logs (app output).
  • When an API fails, you check:
    • EC2 health & CPU
    • Application logs
    • Recent deployments
    • Downstream dependencies
  • For network issues, you check:
    • Security group rules
    • NACL rules
    • Route tables & gateways
Questions & Answers – AWS
Q: An EC2‑hosted API is suddenly unreachable. What do you check first?
A: Security group rules (inbound), instance health, and CloudWatch logs. Then VPC routing and NACLs if needed.
Q: Which AWS service stores logs and metrics for EC2?
A: CloudWatch.
Q: What’s the difference between a user and a role in IAM?
A: A user represents a human; a role represents a machine or service that needs temporary permissions.
AWS Quiz (5 Questions)

1. Which service is used to run virtual machines in AWS?

2. Which AWS service is primarily used for object storage?

3. What is a Security Group?

4. Which service is used to collect logs and metrics from EC2?

5. What is the main difference between an IAM user and an IAM role?

Systems Analyst Scenarios – Real‑World Thinking

This is where you stop being “someone who knows tools” and become “someone people call when things are on fire.” The scenarios below are exactly the kind that show up in assessments and interviews.

Core Scenario Patterns

  • Requirements are unclear or conflicting.
  • An API returns the wrong data or fails intermittently.
  • A production incident hits during peak hours.
  • A sprint story is blocked by a dependency.
  • Two teams want different behaviors from the same system.
Scenario – Unclear Requirements

A product owner gives you a user story: “As a user, I want better search so I can find things faster.” There are no acceptance criteria, and engineering is blocked.

Good Systems Analyst response: You schedule a quick clarification session, ask concrete questions (“What does ‘better’ mean? Filters? Sorting? Response time?”), rewrite the story with clear acceptance criteria, and update Jira so the team can move forward.

Scenario – API Returns Wrong Data

A REST API returns 200 OK, but the JSON payload is missing a field that downstream systems depend on.

Good Systems Analyst response: Reproduce the call in Swagger/Postman, compare the response to the API contract, check recent deployments and logs, and work with engineering to either fix the implementation or update the contract. You document the decision and notify all consumers.

Questions & Answers – Systems Analyst Scenarios

Q: A critical app goes down during peak hours. What’s your first move?
A: Initiate incident triage: confirm the impact, gather logs, check health dashboards, and establish a communication channel. You stabilize first, then analyze root cause.
Q: Two teams want different behaviors from the same API. What do you do?
A: Bring both teams together, document each requirement, map the impact, and propose a unified behavior or versioning strategy. You ensure there is one clear contract for engineering to implement.
Q: A SQL‑based report suddenly runs slowly after a change. How do you approach it?
A: Review the query changes, check JOINs and filters, inspect the execution plan, and look for missing indexes or new many‑to‑many relationships causing row explosion.
Systems Analyst Quiz (5 Questions)

1. A user story is vague and blocking development. What is your best first step?

2. An API returns 200 OK but the payload is missing fields. What do you do?

3. A critical production system goes down. What is the best first action?

4. Two teams want conflicting behaviors from the same API. What should you do?

5. A report shows duplicate rows after a JOIN change. What is the likely cause?

Agile – Scrum, Stories, Acceptance Criteria

Agile isn’t about ceremonies; it’s about feedback loops. The ceremonies just make sure the right conversations happen at the right time. A good Systems Analyst uses Agile to keep ambiguity from sneaking into production.

Scrum Ceremonies & Roles

  • Sprint Planning – Decide what to deliver this sprint and how.
  • Daily Standup – 15‑minute sync: yesterday, today, blockers.
  • Backlog Refinement – Clarify stories, add acceptance criteria, estimate.
  • Sprint Review – Demo completed work to stakeholders.
  • Retrospective – Improve how the team works, not just what it builds.
Example – User Story & Acceptance Criteria
User Story:
  As a customer,
  I want to reset my password via email
  so that I can regain access if I forget it.

Acceptance Criteria:
  Given I am on the login page
  When I click "Forgot Password" and enter a valid email
  Then I receive a reset link within 5 minutes
  And the link expires after 30 minutes
  And using the link lets me set a new password.

This is specific, testable, and unambiguous. Engineering knows what to build; QA knows what to test.

Definition of Ready & Done

  • Definition of Ready (DoR) – A story is ready when:
    • Requirements are clear.
    • Acceptance criteria are defined.
    • Dependencies are identified.
    • It’s estimated and not blocked.
  • Definition of Done (DoD) – A story is done when:
    • Code is complete and tested.
    • Integrated into the main branch.
    • Meets acceptance criteria.
    • Accepted by the Product Owner.
Questions & Answers – Agile
Q: What’s the purpose of backlog refinement?
A: To clarify requirements, define acceptance criteria, estimate work, and ensure stories are ready for development.
Q: A developer is blocked by a dependency. What do you do?
A: Identify the blocker, coordinate with the dependent team, update the story status, and work with the PO to adjust priorities if needed.
Agile Quiz (5 Questions)

1. What is the primary purpose of backlog refinement?

2. Why are acceptance criteria important?

3. What is the purpose of a daily standup?

4. Who defines the Definition of Done?

5. A story is not ready (unclear, unestimated). What should happen?

Java & Spring – How the Engine Actually Works

You don’t need to be the mechanic turning every bolt, but you do need to know how the engine is laid out. Spring is the wiring harness that connects everything; Java is the metal. Once you see the pattern, the jargon stops being scary.

Spring Core & Dependency Injection

  • Spring Core – Provides dependency injection (DI) and inversion of control (IoC).
  • Bean – An object managed by the Spring container.
  • Dependency Injection – Spring creates and injects dependencies instead of you doing new everywhere.
Example – Service & Controller
@Service
public class OrderService {
  public Order createOrder(OrderRequest request) {
    // business logic here
  }
}

@RestController
@RequestMapping("/orders")
public class OrderController {

  private final OrderService orderService;

  public OrderController(OrderService orderService) {
    this.orderService = orderService; // injected by Spring
  }

  @PostMapping
  public ResponseEntity<Order> create(@RequestBody OrderRequest request) {
    Order order = orderService.createOrder(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(order);
  }
}

The controller handles HTTP, the service handles business logic, and Spring wires them together. As a Systems Analyst, you don’t need to write this, but you should recognize the pattern when debugging.

REST Controllers, Exceptions & JSON

  • @RestController – Combines @Controller and @ResponseBody; returns JSON/XML directly.
  • Exception Handling – Try/catch blocks and global handlers keep APIs from crashing.
  • JSON Serialization – Spring converts Java objects to JSON and back automatically.
Questions & Answers – Java & Spring
Q: What is dependency injection in Spring?
A: It’s a pattern where Spring creates and provides the objects a class needs, instead of the class constructing them itself. This reduces coupling and improves testability.
Q: What does @RestController do?
A: It marks a class as a REST controller whose methods return data (JSON/XML) directly as HTTP responses.
